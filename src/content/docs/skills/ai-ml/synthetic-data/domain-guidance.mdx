## What You Can Build

You can generate synthetic data that matches the statistical fingerprint of a specific industry -- e-commerce order funnels with cart abandonment, support tickets with correlated resolution times and CSAT scores, IoT sensor readings with age-based degradation, or financial transactions with fraud indicators. Each domain has characteristic distributions, table relationships, and edge cases that make the data useful for demos and pipeline testing.

## In Action

> "Generate e-commerce order amounts that vary by customer tier using log-normal distributions. Premium customers should spend significantly more than Basic. Use Python with PySpark."

```python
from pyspark.sql import functions as F
from pyspark.sql.types import DoubleType
import pandas as pd
import numpy as np

@F.pandas_udf(DoubleType())
def generate_order_amount(tiers: pd.Series) -> pd.Series:
    """E-commerce order amounts by tier."""
    amounts = []
    for tier in tiers:
        if tier == "Premium":
            amounts.append(float(np.random.lognormal(mean=5.5, sigma=0.9)))   # ~$245 median
        elif tier == "Standard":
            amounts.append(float(np.random.lognormal(mean=4.2, sigma=0.7)))   # ~$67 median
        else:  # Basic
            amounts.append(float(np.random.lognormal(mean=3.5, sigma=0.6)))   # ~$33 median
    return pd.Series(amounts)

# Order status with realistic cart abandonment
orders_df = orders_df.withColumn("amount", generate_order_amount(F.col("tier")))
orders_df = orders_df.withColumn("status",
    F.when(F.rand() < 0.70, "abandoned")
     .when(F.rand() < 0.85, "pending")
     .when(F.rand() < 0.92, "processing")
     .when(F.rand() < 0.97, "shipped")
     .otherwise("delivered")
)
```

**Key decisions:**
- **Log-normal per tier** produces the long tail real revenue data has -- most orders are modest, some are large. The tier parameter shifts the entire distribution, not just the cap.
- **70% abandoned** reflects real e-commerce funnels where most carts never complete. This matters for pipeline testing -- dashboards that only handle "delivered" orders will break.
- **Table chain: customers, orders, order_items, products** -- generate in this order, writing master tables to Delta first, then reading back for FK joins.

### Support Tickets with Correlated Metrics

> "Generate support tickets where priority, resolution time, and CSAT scores are correlated with customer tier. Enterprise customers should get faster resolution. Use Python with PySpark."

```python
@F.pandas_udf("struct<priority:string,resolution_hours:double,csat:int>")
def generate_ticket_metrics(tiers: pd.Series) -> pd.DataFrame:
    """Support ticket metrics with correlated attributes."""
    results = []
    for tier in tiers:
        if tier == 'Enterprise':
            priority = np.random.choice(['Critical', 'High', 'Medium'], p=[0.3, 0.5, 0.2])
        else:
            priority = np.random.choice(['Critical', 'High', 'Medium', 'Low'], p=[0.05, 0.2, 0.45, 0.3])

        resolution_scale = {'Critical': 4, 'High': 12, 'Medium': 36, 'Low': 72}
```

### IoT Sensor Readings with Degradation

> "Generate time-series sensor readings where older equipment shows drift and more frequent anomalies. Use Python with PySpark."

```python
@F.pandas_udf(DoubleType())
def generate_sensor_reading(equipment_ages: pd.Series) -> pd.Series:
    """Sensor readings with age-based degradation."""
    readings = []
    for age_days in equipment_ages:
        base = 100.0
        drift = (age_days / 365) * 5          # 5 units drift per year
        noise = np.random.normal(0, 2)

        anomaly_prob = min(0.01 + (age_days / 365) * 0.02, 0.1)
        if np.random.random() < anomaly_prob:
            noise += np.random.choice([-1, 1]) * np.random.exponential(10)

        readings.append(base + drift + noise)
    return pd.Series(readings)
```

The drift increases linearly with equipment age, and anomaly frequency scales from 1% (new equipment) to 10% (old equipment). This pattern produces data that predictive maintenance models can actually learn from. Table chain: equipment, sensors, readings, maintenance_orders.

### Financial Transactions with Fraud Indicators

> "Generate transaction amounts following a power law by account type, with fraud flags correlated to amount and time of day. Use Python with PySpark."

```python
@F.pandas_udf(DoubleType())
def generate_transaction_amount(account_types: pd.Series) -> pd.Series:
    """Transaction amounts following power law by account type."""
    amounts = []
    for acct_type in account_types:
        if acct_type == "Corporate":
            amount = (np.random.pareto(a=1.5) + 1) * 1000
        elif acct_type == "Premium":
            amount = np.random.lognormal(mean=6, sigma=1.2)
        else:
            amount = np.random.lognormal(mean=4, sigma=0.8)
        amounts.append(min(amount, 1_000_000))
    return pd.Series(amounts)

@F.pandas_udf(BooleanType())
def generate_fraud_flag(amounts: pd.Series, hours: pd.Series) -> pd.Series:
    """Flag suspicious transactions based on amount and time."""
    flags = []
    for amount, hour in zip(amounts, hours):
        base_prob = 0.001
        if amount > 5000:
            base_prob *= 3
        if hour < 6 or hour > 22:
            base_prob *= 2
        flags.append(np.random.random() < base_prob)
    return pd.Series(flags)
```

Corporate accounts use Pareto distribution (power law) because real corporate spending has extreme outliers. Fraud probability correlates with two signals: large amounts and unusual hours. This gives ML models real signal to learn from. Table chain: accounts, transactions, payments, fraud_flags.

## Watch Out For

- **Flat distributions in domain data** -- every domain has characteristic shapes. E-commerce amounts are log-normal, IoT readings are normal with drift, financial transactions follow power laws. Using `F.rand() * 1000` for any of these produces obviously fake data.
- **Independent attributes** -- real data has correlations. Support ticket priority should correlate with customer tier, CSAT should correlate with resolution time, fraud probability should correlate with amount. Generate them together, not independently.
- **Missing the table chain** -- generate master tables first (customers, accounts, equipment), write to Delta, read back, then generate child tables with valid FK joins. Generating FK values with random IDs breaks referential integrity.
- **Skipping edge cases** -- real datasets have nulls, anomalies, and outliers. Inject 1-3% null values in optional fields and add anomaly logic to time-series data to make the synthetic data behave like production data in error-handling code.
