AI/BI Dashboards provide a low-code way to build data visualizations, KPI displays, and filtered reports on top of Unity Catalog tables. Use the dashboard JSON API when you need programmatic control over datasets, widgets, layouts, and filters -- for example, generating dashboards from templates or deploying them across workspaces via CI/CD.

## Overview

The AI/BI Dashboards skill teaches Claude Code how to create, configure, and deploy Databricks AI/BI dashboards (formerly Lakeview dashboards) using the dashboard JSON API. Use it when building data visualizations, KPI displays, or interactive filtered reports directly from Unity Catalog tables. It covers dataset design, widget configuration, layout rules, and filter architecture.

## Capabilities

- Build complete dashboards with datasets, widgets, and multi-page layouts
- Create KPI counters, bar charts, line charts, pie charts, and data tables
- Design global filters (cross-dashboard) and page-level filters
- Validate SQL queries against warehouses before deploying dashboard JSON
- Configure 6-column grid layouts with proper widget sizing and spacing
- Use pre-aggregated and aggregating widget patterns for counters
- Apply Spark SQL patterns for date math, truncation, and grouping
- Publish and unpublish dashboards for sharing

## Key Concepts

### Dashboard JSON Structure

Dashboards are defined as JSON with three main sections:
- **Datasets** — SQL queries that feed widgets, each with a unique name and fully-qualified table references
- **Pages** — Canvas or global-filter pages containing widget layouts
- **Widgets** — Visual elements (counters, charts, tables, filters, text) positioned on a 6-column grid

### 6-Column Grid Layout

Every dashboard uses a 6-column grid. Each widget has a position (`x`, `y`, `width`, `height`) and every row must sum to exactly width 6 with no gaps.

| Widget Type | Recommended Width | Recommended Height |
|-------------|------------------|--------------------|
| Text header | 6 | 1 |
| Counter/KPI | 2 | 3-4 |
| Line/Bar chart | 3 | 5-6 |
| Pie chart | 3 | 5-6 |
| Full-width chart | 6 | 5-7 |
| Table | 6 | 5-8 |

### Widget Version Requirements

Different widget types require specific version numbers in their spec:

| Widget Type | Version |
|-------------|---------|
| counter | 2 |
| table | 2 |
| filter-multi-select | 2 |
| filter-single-select | 2 |
| filter-date-range-picker | 2 |
| bar | 3 |
| line | 3 |
| pie | 3 |
| text | N/A (no spec block) |

### Field Name Matching

The `name` in `query.fields` must exactly match the `fieldName` in widget `encodings`. A mismatch causes "no selected fields to visualize" errors.

```json
// Correct: names match
"fields": [{"name": "sum(spend)", "expression": "SUM(`spend`)"}]
"encodings": {"value": {"fieldName": "sum(spend)", "displayName": "Total Spend"}}
```

### Global vs Page-Level Filters

| Type | Placement | Scope |
|------|-----------|-------|
| **Global** | Dedicated page with `PAGE_TYPE_GLOBAL_FILTERS` | Affects all pages with matching dataset fields |
| **Page-level** | Regular canvas page | Affects only widgets on that page |

Filter widget types: `filter-multi-select`, `filter-single-select`, `filter-date-range-picker`. Always use `spec.version: 2` and include `frame` with `showTitle: true`.

### Dataset Architecture

- One dataset per domain (orders, customers, products)
- Exactly one valid SQL query per dataset (no semicolons)
- Always use fully-qualified table names: `catalog.schema.table_name`
- Put all business logic (CASE/WHEN, COALESCE, ratios) in the dataset SQL with explicit aliases
- Every widget `fieldName` must match a dataset column or alias exactly

### Cardinality Guidelines

Charts with color/grouping dimensions should have 3-8 distinct values. High-cardinality dimensions (customer IDs, order IDs) should use table widgets instead of charts. For dimensions with more than 10 values, aggregate to a higher level or use a TOP-N + "Other" bucket pattern.
