## What You Can Build

AI/BI Dashboards give you programmatic control over data visualizations, KPI counters, charts, tables, and filtered reports on top of Unity Catalog tables. You can generate dashboards from templates, deploy them across workspaces via CI/CD, and configure multi-page layouts with global and page-level filters -- all through a JSON API.

## In Action

> "Build a dashboard with KPI counters for total trips, average fare, and average distance, plus a bar chart of trips by pickup ZIP code using the NYC taxi dataset."

```python
import json

# Test queries first (mandatory validation step)
execute_sql("SELECT COUNT(*) as trips, AVG(fare_amount) as avg_fare, "
            "AVG(trip_distance) as avg_distance FROM samples.nyctaxi.trips")

execute_sql("SELECT pickup_zip, COUNT(*) as trip_count "
            "FROM samples.nyctaxi.trips GROUP BY pickup_zip "
            "ORDER BY trip_count DESC LIMIT 10")

# Build dashboard JSON
dashboard = {
    "datasets": [
        {"name": "summary", "displayName": "Summary Stats",
         "queryLines": ["SELECT COUNT(*) as trips, AVG(fare_amount) as avg_fare, ",
                        "AVG(trip_distance) as avg_distance FROM samples.nyctaxi.trips"]},
        {"name": "by_zip", "displayName": "Trips by ZIP",
         "queryLines": ["SELECT pickup_zip, COUNT(*) as trip_count ",
                        "FROM samples.nyctaxi.trips GROUP BY pickup_zip ",
                        "ORDER BY trip_count DESC LIMIT 10"]}
    ],
    "pages": [{"name": "overview", "displayName": "Overview",
               "pageType": "PAGE_TYPE_CANVAS", "layout": [...]}]
}

result = create_or_update_dashboard(
    display_name="NYC Taxi Dashboard",
    serialized_dashboard=json.dumps(dashboard),
    warehouse_id=get_best_warehouse()
)
```

**Key decisions:**
- Always test SQL queries with `execute_sql` before embedding them in dashboard JSON -- untested queries produce "Invalid widget definition" errors
- One dataset per domain (orders, customers, products) with fully qualified table names
- The 6-column grid layout requires every row to sum to exactly width 6 with no gaps
- Counters use `spec.version: 2`, bar/line/pie charts use `spec.version: 3`
- Every widget `fieldName` in encodings must exactly match the `name` in query fields

## More Patterns

### Add global filters that affect all pages

> "Build a dashboard with a global region filter that slices all widgets across every page."

```python
# Global filter page
{
    "name": "filters",
    "displayName": "Filters",
    "pageType": "PAGE_TYPE_GLOBAL_FILTERS",
    "layout": [{
        "widget": {
            "name": "filter_region",
            "queries": [{"name": "ds_data_region", "query": {
                "datasetName": "sales",
                "fields": [{"name": "region", "expression": "`region`"}],
                "disaggregated": False
            }}],
            "spec": {
                "version": 2,
                "widgetType": "filter-multi-select",
                "encodings": {"fields": [{"fieldName": "region",
                    "displayName": "Region", "queryName": "ds_data_region"}]},
                "frame": {"showTitle": True, "title": "Region"}
            }
        },
        "position": {"x": 0, "y": 0, "width": 2, "height": 2}
    }]
}
```

Global filters live on a dedicated page with `PAGE_TYPE_GLOBAL_FILTERS` and affect all pages that have datasets containing the filter field. Page-level filters go on regular canvas pages and affect only that page's widgets. Filter types: `filter-multi-select`, `filter-single-select`, `filter-date-range-picker`.

### Design a standard dashboard layout

> "Show me the recommended widget layout for a dashboard with KPIs, charts, and a detail table."

The 6-column grid follows a standard structure:

```text
y=0:  Title text widget (w=6, h=1)
y=1:  Subtitle text widget (w=6, h=1)
y=2:  Three KPI counters (w=2 each, h=3)
y=5:  Section header (w=6, h=1)
y=6:  Two side-by-side charts (w=3 each, h=5-6)
y=12: Full-width data table (w=6, h=5-8)
```

Text widgets use `multilineTextboxSpec` with no `spec` block. Title and subtitle must be separate widgets at different y positions -- putting them in the same `lines` array concatenates them on one line.

### Configure counter widgets for KPIs

> "Show the two patterns for counter widgets: pre-aggregated (single row) and aggregating (multi-row with filters)."

```python
# Pattern 1: Pre-aggregated dataset (1 row, no filters)
{"name": "revenue", "expression": "`revenue`"}  # field
{"fieldName": "revenue", "displayName": "Total Revenue"}  # encoding
# Use disaggregated: True

# Pattern 2: Aggregating widget (multi-row, supports filters)
{"name": "sum(spend)", "expression": "SUM(`spend`)"}  # field
{"fieldName": "sum(spend)", "displayName": "Total Spend"}  # encoding
# Use disaggregated: False
```

The field `name` and encoding `fieldName` must match exactly. For aggregating counters, the convention is to use the aggregation expression as the name (e.g., `"sum(spend)"`). Percent values must be 0-1 in the data, not 0-100.

## Watch Out For

- **Deploying without testing queries via `execute_sql`** -- untested queries are the top cause of broken dashboard widgets. Test every dataset query and verify column names match widget field references before deploying.
- **Mismatching field names between query fields and encodings** -- `name` in `query.fields` must exactly match `fieldName` in widget `encodings`. A mismatch produces "no selected fields to visualize" errors. For aggregations, use the same string in both places (e.g., `"sum(spend)"`).
- **Using wrong version numbers for widget types** -- counters, tables, and filters need `version: 2`. Bar, line, and pie charts need `version: 3`. Text widgets have no `spec` block at all. Wrong versions produce "Invalid widget definition" errors.
- **Charting high-cardinality dimensions** -- charts with color/grouping dimensions should have 3-8 distinct values. For dimensions with 10+ values, use a TOP-N with "Other" bucket pattern, or switch to a table widget.
- **Using `widgetType: "filter"` for filter widgets** -- this type doesn't exist. Use `filter-multi-select`, `filter-single-select`, or `filter-date-range-picker` instead.
