External APIs fail -- network timeouts, rate limits, transient server errors. This page provides production-ready patterns for exponential backoff, circuit breakers, dead letter queues, and graceful degradation so your custom Spark data source recovers automatically instead of crashing the job.

## Exponential Backoff

```python
def write_with_retry(self, iterator):
    for row in iterator:
        for retry_count in range(self.max_retries + 1):
            try:
                self._send_data(row)
                break
            except Exception as e:
                if not self._is_retryable_error(e):
                    raise
                if retry_count >= self.max_retries:
                    raise Exception(f"Max retries exceeded: {e}")
                backoff = min(1.0 * (2 ** retry_count), 60.0)
                time.sleep(backoff)

def _is_retryable_error(self, error):
    if isinstance(error, (Timeout, ConnectionError)):
        return True
    if hasattr(error, 'response') and error.response:
        return error.response.status_code in (429, 500, 502, 503, 504)
    return False
```

## Circuit Breaker

```python
class CircuitBreaker:
    def __init__(self, threshold=10, timeout=300):
        self.threshold = threshold
        self.timeout = timeout
        self.consecutive_failures = 0
        self.circuit_open = False

    def record_success(self):
        self.consecutive_failures = 0

    def record_failure(self):
        self.consecutive_failures += 1
        if self.consecutive_failures >= self.threshold:
            self.circuit_open = True
            self.circuit_open_until = datetime.now() + timedelta(seconds=self.timeout)
```

## Dead Letter Queue

```python
class DeadLetterQueueWriter:
    def write(self, iterator):
        for row in iterator:
            try:
                self._send_data(row)
            except Exception as e:
                if self.dlq_enabled:
                    self._write_to_dlq(row, e)
                else:
                    raise
```

## Retry with Throttling Respect

Handle `Retry-After` headers from rate-limited APIs:

```python
def _get_retry_after(self, response):
    retry_after = response.headers.get("Retry-After")
    if retry_after:
        try:
            return int(retry_after)
        except ValueError:
            pass
    return 1.0  # Default fallback
```

## Best Practices

1. **Retry only transient errors** -- never retry client errors (4xx except 429)
2. **Respect rate limits** -- use Retry-After headers
3. **Circuit breakers** -- prevent cascading failures
4. **Timeout operations** -- set reasonable timeouts
5. **Dead letter queues** -- store failed records for later analysis
6. **Monitor failure rates** -- alert on anomalous error rates
